🗺️ System Architecture Overview

The application is built on the MERN stack (MongoDB, Express, React, Node.js). It adheres to a clean, client-server decoupled design model where the client communicates with the server via structured HTTP REST APIs.

🛠️ The Tech Stack (What & Why)

1. The Frontend (Client-Side)

  • React & Vite: We chose React for its component-driven design and virtual DOM, making UI state changes extremely smooth. We paired it with Vite because it offers extremely fast reload times (HMR) and uses native ES modules, outperforming older tools like Create React App.
  • Tailwind CSS (v4): Provides a utility-first styling system. It enables custom color matching, responsive layouts (mobile and desktop support), and transitions without writing bloated custom CSS stylesheets.
  • Recharts: Used on the Dashboard overview. It renders vector (SVG) charts that dynamically scale based on user placement analytics (e.g., Application success ratios, DSA metrics).

2. The Backend (Server-Side)

  • Node.js & Express.js: Node.js handles high-performance, asynchronous non-blocking operations. Express.js sits on top to structure routes and manage middleware (like verifying JSON tokens and mapping requests).
  • Mongoose ODM: Maps JS objects directly to database schemas. It enforces validation, structures relations (like embedding interview rounds into company documents), and exposes connection helpers.
  • Bcrypt.js: Securely hashes user passwords on signup using a strong algorithm with salt. Securely validates passwords during login without ever saving clean password strings.

🗄️ Database Schemas & Data Model Specifications

1. User Authentication Model

Manages login capabilities. It includes a custom pre-save function to encrypt passwords before saving to database storage.

// Pre-save hook: Hash password securely
userSchema.pre('save', async function() {
  if (!this.isModified('password')) return;
  const salt = await bcrypt.genSalt(10);
  this.password = await bcrypt.hash(this.password, salt);
});

2. Job Applications Model

Tracks target companies, dates, status, notes, and individual interview rounds. The rounds are defined as nested subdocuments in a child schema:

const roundSchema = new mongoose.Schema({
  name: {
    type: String,
    required: true,
    enum: ['Aptitude', 'Group Discussion', 'Technical Interview', 'HR Interview', 
           'Communication Round', 'DSA Round', 'Online Assessment', 'Other'],
  },
  status: {
    type: String,
    required: true,
    enum: ['Pending', 'Cleared', 'Not Cleared'],
    default: 'Pending',
  },
  date: Date
});

🎓 Common Interview Questions & Answers

Q1 "How does the database connection fallback function in your backend server?"

In server.js, we designed the connection helper to handle environment issues. It first tries connecting to a standard local MongoDB. If that fails (meaning the local MongoDB service isn't running), it launches a temporary mongodb-memory-server in-memory instance, connects to it, and runs a data-seeding script automatically. This guarantees that our project runs out-of-the-box anywhere without database configuration dependencies.

Q2 "Explain how authentication security works in this app."

We use token-based authentication (JWT). When a user registers or logs in, the backend encrypts/verifies the password using bcryptjs, generates a JSON Web Token signed with a backend secret key, and sends it to the client. The frontend stores this token and appends it to the Authorization: Bearer <token> header for any protected HTTP requests. Our custom authentication middleware intercepts the route, validates the token, and attaches user info to the request.

Q3 "What is CORS and why did you need it?"

CORS stands for Cross-Origin Resource Sharing. Browsers restrict client websites from querying servers on a different domain. Since our frontend is hosted on port 5173 (Vite) and our API runs on port 5000, requests would get blocked by the browser. By calling app.use(cors()) in Express, the backend sends authorization headers back, signaling to the browser that it trusts and welcomes calls from our React origin.

Q4 "How do you handle nested relations like interview rounds in MongoDB?"

Instead of creating two separate database collections for "Companies" and "Interview Rounds" (which would require joins), we model rounds as an array of subdocuments directly nested inside the main Company schema. This is a best-practice document design in MongoDB because interview rounds are always queried and displayed in relation to their company, resulting in fast single-document lookups.
